# AGENTS.md — Advent of Code Java Project: AI Agent Guide



## Project Architecture & Data Flow
- **Monorepo for Advent of Code (AoC) solutions**: Each puzzle is a standalone Java class, grouped by year and day under `odogwudozilla.year<YYYY>.day<D>`.
- **Resource-driven**: All puzzle input and descriptions are stored in `src/main/resources/<YYYY>/day<D>/` as text files. No hardcoded puzzle data.
- **Puzzle metadata and progress**: Managed in `aoc_challenge_config.json` (available puzzles) and `solutions_database.json` (solved puzzles/results).
- **Enhancement features**: Core utilities, dashboard, and statistics live under `src/main/java/odogwudozilla/core/`, `dashboard/`, and `docs/enhancements/`.

### Enhancement Feature Modules
- `src/main/java/odogwudozilla/core/PuzzleTestHarness.java`: Test harness for validating puzzle outputs (expected vs. actual, formatted reports)
- `src/main/java/odogwudozilla/core/PuzzlePerformanceMonitor.java`: Tracks execution time and memory usage for each puzzle part
- `src/main/java/odogwudozilla/core/PuzzleCacheManager.java`: Caches puzzle results for quick comparison and regression checks
- `src/main/java/odogwudozilla/core/PuzzleDifficultyRater.java`: Rates puzzle difficulty based on time, memory, and code size
- `src/main/java/odogwudozilla/core/SolutionStatisticsCollector.java`: Aggregates and reports statistics across all solutions
- `src/main/java/odogwudozilla/dashboard/WebDashboardGenerator.java`: Generates the interactive HTML dashboard from solution data (see `dashboard/index.html`).

All enhancement modules are integrated and referenced in both the codebase and documentation. See `docs/enhancements/ENHANCEMENT_FEATURES.md` for details and usage examples. Example integration: `src/main/java/odogwudozilla/examples/EnhancementFeaturesExample.java`.

#### Cache Directory
All cached puzzle results are stored in `cache/puzzle-results/`. Agents must use this directory for all cache operations.

#### Dashboard Generation
The interactive dashboard is generated by `WebDashboardGenerator.java` and outputs to `dashboard/index.html`. Agents should not manually edit the dashboard file; always regenerate via the enhancement module.


## Key Developer Workflows
- **Run a specific puzzle**: `puzzle <year> day<day>` (preferred) or `./gradlew run --args="<year> day<day>"`
- **Random unsolved puzzle (select only)**: `./gradlew randomPuzzle`
- **Install Playwright browsers** (one-time): `./gradlew installPlaywright`
- **Build project**: `./gradlew build`
- **Test**: `./gradlew test`
- **Run enhancement example**: `./gradlew runEnhancementExample`

### Automation Pipeline and Chat Triggers

| Command | Effect | Automated end-to-end? |
|---------|--------|----------------------|
| `./gradlew autoSolve` | Full pipeline; pauses at Stage 5 for user to implement | No - user solves |
| `./gradlew autoSolve --args="--setup --watch"` | Stages 1-4: fetch, scrape, generate skeleton only | Partial |
| `./gradlew autoSolve --args="--auto --watch"` | Full pipeline, skip pause (solution already written) | Yes, if already implemented |
| `./gradlew autoSolve --args="--submit YEAR DAY --watch"` | Stages 6-12 for a specific puzzle | Yes, if already implemented |

**Chat triggers:** Typing `random` or `solve` in chat will:
1. Check for `.aoc-state` (pending puzzle) and resume if found
2. If not found, run `--setup --watch` to fetch and generate skeleton
3. Read the description file and implement solve methods
4. Run `--auto --watch` to submit, document, and commit
5. All chat-driven commands always include `--watch` for browser visibility

**No hardcoded input:** All solutions must read from resource files. See `.github/copilot-instructions.md` for full workflow and automation details.


## Project-Specific Conventions
- **Class naming**: `<PuzzleTitle>AOC<YYYY>Day<D>.java` (e.g., `SecretEntranceAOC2025Day1.java`)
- **Package structure**: `odogwudozilla.year<YYYY>.day<D>`
- **Resource files**: Always use `day<D>_puzzle_description.txt` and `day<D>_puzzle_data.txt` for each puzzle
- **No hardcoded input**: Always read from resource files
- **Single-class per puzzle**: Unless complexity requires otherwise
- **JavaDocs**: Must include puzzle summary and official URL
- **No wildcard imports**; use JetBrains annotations (`@NotNull`, `@Nullable`)
- **British English for comments/JavaDocs**
- **No magic numbers**: Use named constants

### Additional Conventions
- **Cache directory**: All cached results must be written to `cache/puzzle-results/`.
- **Dashboard generation**: The dashboard is always generated by `WebDashboardGenerator.java` and outputs to `dashboard/index.html`.
- **Enhancement modules**: All six core modules are required for solution validation, performance, caching, rating, statistics, and dashboard generation. See `src/main/java/odogwudozilla/core/` and `docs/enhancements/ENHANCEMENT_FEATURES.md`.
- **No wildcard imports**: Always use explicit imports. Use JetBrains annotations where appropriate.
- **British English**: All comments and JavaDocs must use British English, even if other code is in Dutch.

### Hybrid/Polyglot Solutions
- Some puzzles (e.g., 2025 Day 10) require a hybrid approach:
  - Part 1: Java (main solution class)
  - Part 2: Python (Z3 solver, see `solve_part2_z3.py` in the puzzle directory)
- See the puzzle README for instructions on running both parts and requirements (e.g., `z3-solver` Python package).


## Integration Points & Automation

### Automation Pipeline (`odogwudozilla.automation` package)
Full end-to-end automation is triggered via `./gradlew autoSolve`. See `docs/AOC_AUTOMATION_PLAN.md` for full design details.

| Class | Responsibility |
|-------|---------------|
| `AutomationConfig` | All constants - URLs, paths, env vars, signal strings |
| `BrowserSessionManager` | Playwright lifecycle + AoC session cookie injection |
| `PuzzleScraper` | Scrapes puzzle title and description from AoC pages |
| `InputFetcher` | Fetches puzzle input and writes to resources folder |
| `SolutionSkeletonGenerator` | Creates Java class stub and resource text files |
| `SolverRunner` | Runs puzzle via Gradle, parses `Part 1:`/`Part 2:` stdout |
| `AnswerSubmitter` | Submits answers via Playwright form POST |
| `AcceptanceVerifier` | Parses AoC response for correct/incorrect/too-soon signals |
| `RateLimiter` | Enforces >= 60 s delay between submissions |
| `DocumentationUpdater` | Updates `solutions_database.json` and README files |
| `GitAutomationService` | Runs `git add .` and `git commit` |
| `PuzzleInfo` | Immutable data holder for puzzle metadata |
| `SubmissionResult` | Enum: CORRECT, INCORRECT, TOO_SOON, ALREADY_SOLVED, UNKNOWN |

Main orchestrator: `src/main/java/odogwudozilla/AutomationOrchestrator.java`

### Session Authentication
- Set env var `AOC_SESSION` or create `.aoc-session` in project root
- Never commit session values; `.aoc-session` is gitignored

### Other Automation
- **PuzzleRandomizer**: Selects unsolved puzzles using a two-tier priority:
  1. If any year has zero solved days, randomly picks one of those years and returns day 1
  2. Otherwise, randomly selects a year and returns the lowest-numbered unsolved day from it
  - Never selects Day 25 unless all days 1-24 for that year are solved
- **Dashboard**: Progress and statistics in `dashboard/index.html` (auto-generated by `WebDashboardGenerator.java`)
- **Yearly and main README**: Updated automatically by `DocumentationUpdater`
- **Cache**: JSON results in `cache/puzzle-results/`
- **Test Harness/Performance/Stats**: Use all six core enhancement modules for validation, timing, caching, rating, statistics, and dashboard reporting (see above)

### How `puzzle` / `Main.java` Discovers Classes
`Main.java` uses reflection to locate the puzzle class at runtime. It scans the compiled package directory `odogwudozilla.year<YYYY>.day<D>`, finds the `.class` file whose name contains the suffix `AOC<YYYY>Day<D>` (excluding inner classes ending in `$`), then invokes its `public static main(String[] args)` method. The project must be built (`./gradlew build`) before running a puzzle.

## Special Cases
- **Day 25**: Part 2 is locked until all previous days are solved; randomizer skips Day 25 unless eligible
- **Do not run solution classes directly**: Always use Gradle tasks or the `puzzle` command for correct environment
